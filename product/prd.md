# 一、产品基础信息

## 1.1 产品定位

一个以 AI 为核心驱动的健康生活管理 App，帮助用户建立可持续的饮食、运动与作息习惯。

关键词：

* AI驱动
* 健康管理
* 个性化节律
* 长期习惯养成

---

## 1.2 产品目标

### 短期目标

帮助用户：

* 建立运动习惯
* 建立饮食管理习惯
* 建立基础作息节律

### 长期目标（产品内核，不对外表达）

构建用户长期健康行为模型，实现动态优化。

---

# 二、核心产品理念（内核）

1. 首页不是打卡中心，而是状态中心
2. AI 是决策中枢，而不是对话装饰
3. 任务数量最小化，行动质量最大化
4. 数据记录为“输入源”，不是“产品主角”
5. 长期数据沉淀是产品资产

---

# 三、目标用户

城市年轻职场人
20-35岁
具备健康意识但难以长期坚持

---

# 四、产品核心能力结构

    ┌────────────────┼────────────────┐

# 2️⃣ AI伙伴页

核心交互入口，用户与 AI 进行自然语言对话，AI 根据意图自动调用后端能力完成操作或回答问题。

---

## 功能

* 对话交互（流式输出，逐字显示）
* 计划调整（生成/修改训练计划、饮食计划）
* 情绪支持与动力激励
* 周期复盘（结合历史数据分析）
* 目标设定与调整
* 数据查询（"我今天吃了多少热量？"）
* 健康知识问答

---

## 交互设计

### 页面结构

```
┌────────────────────────┐
│ 顶部：对话标题 / 历史入口  │
├────────────────────────┤
│                        │
│   对话消息列表           │
│   （支持文字 + 卡片）     │
│                        │
├────────────────────────┤
│ AI 正在思考...（流式态）  │
├────────────────────────┤
│ 输入框 + 发送按钮        │
└────────────────────────┘
```

### 消息类型

* **文字消息**：普通问答，流式逐字输出
* **操作结果卡片**：AI 执行工具调用后，以卡片形式展示结果（如生成的训练计划、今日饮食汇总）
* **确认卡片**：AI 执行写操作前（如创建记录、修改计划），展示确认按钮，等待用户确认（Human-in-the-loop）

### 流式输出体验

* AI 回复开始时立即显示光标动画
* token 逐字追加，不等待完整响应
* 工具调用期间显示"正在查询数据..."等中间状态提示
* 工具调用完成后继续流式输出最终回答

---

## AI 能力边界

### 可执行操作（通过 Function Calling）

* 查询今日 / 本周饮食记录与热量
* 查询运动记录与消耗
* 查询睡眠数据
* 生成个性化训练计划
* 生成饮食建议
* 创建运动记录
* 查询健康趋势数据

### 知识问答（RAG，后续迭代）

* 营养知识（食物热量、营养素作用）
* 运动知识（动作要领、训练原则）
* 健康常识

### 明确不做

* 不提供医疗诊断建议
* 不回答与健康无关的话题
* 不执行未经用户确认的写操作

---

## AI必须具备能力

* 根据行为数据调整强度
* 识别连续未执行情况
* 预测用户动力波动
* 推荐适合运动类型
* 理解上下文，支持多轮追问（"那换成低强度的呢？"）

# 4️⃣ 饮食系统

## 4.1 目标

降低饮食记录成本，用「拍照 + 估算」替代纯手输；在不确定时允许用户快速存档，后续再校准克数与热量。

## 4.2 核心能力

* **营养价值表识别**（优先模式：直接识别包装食品营养成分表）
* 拍照识别（食物识别/分割/候选菜品）
* 热量估算（支持多食物、多份量）
* 营养分布分析（蛋白/脂肪/碳水与膳食纤维等，可逐步扩展）
* 校准机制（输入克数/份数后重算热量）
* 快捷模式（仅拍照存档，稍后补录克数/选择食物）
* 参考物对照（用于体积/份量估算的稳定参照）

## 4.3 关键交互与流程

### A. 营养价值表识别（优先推荐）

```
记录页 → 饮食 → 拍照
→ 拍摄包装食品的营养成分表
→ OCR 识别营养价值表（热量/蛋白质/脂肪/碳水等）
→ 用户输入实际食用克数
→ 自动计算实际摄入热量与营养素
→ 保存 → 进入当日汇总/趋势
```

**适用场景**：包装食品、有营养标签的食物
**优势**：数据准确、无需估算、录入快速

### B. 参考物拍照记录（标准模式）

```
记录页 → 饮食 → 拍照
→（首次）参考物录入/标定（拳头 + 硬币，仅需一次）
→ 拍照（要求同时拍到食物 + 拳头）
→ 识别结果页（候选食物 + 估算热量 + 置信度）
→ 用户确认/修改（增删食物、选择候选、输入克数/份数）
→ 保存 → 进入当日汇总/趋势
```

**适用场景**：自制菜品、餐厅食物、无包装食品
**关键**：必须同框拍到食物和拳头，用于份量估算

### C. 快捷模式（快速存档）

目标：减少录入耗时，先把证据存下，之后再补齐数据。

```
记录页 → 饮食 → 快捷拍照
→ 仅上传/保存照片（可选备注：早餐/午餐/晚餐/加餐）
→ 记录入库（状态：pending）
→ 稍后在记录详情补录：食物项 + 克数/份数 → 重算热量 → 状态转为 confirmed
```

### D. 参考物/对照物机制（用于份量估算）

* 首次使用参考物拍照估算时，引导用户录入参考物（仅需一次）：
  * **拳头**：拍摄握拳照片，用于大份量食物对比
  * **硬币**：拍摄硬币照片并填写直径（如1元硬币25mm），用于小份量食物对比
* 每次进行「参考物拍照算热量」时，提示用户**同框拍到食物 + 拳头**，以提升份量估算稳定性。

> 说明：参考物用于提升估算精度；用户拒绝/无法提供时仍可继续，但系统需降低置信度并提示用户补录克数以校准。

## 4.4 识别结果页（信息结构）

* 顶部：原图预览 + 重拍
* 识别摘要：总热量（kcal）+ 置信度（高/中/低）+ 「建议校准」提示
* 食物列表（可编辑）：
  * 食物名称（可搜索替换）
  * 份量（克数/份数，可为空）
  * 单项热量（估算/校准后）
  * 删除/新增
* 底部：保存 / 进入快捷模式（当识别不稳定时）

## 4.5 功能边界与异常处理（必须写清）

* **场景限制**：
  * 光线过暗/反光严重、食物遮挡、同框多个盘子过多时，识别可能失败或置信度低
  * 包装食品：优先支持识别「营养成分表/热量表」；否则按食物外观估算（准确性更低）
* **失败兜底**：
  * 识别失败 → 自动进入快捷模式（照片存档）+ 提示用户后续手动补录
  * 置信度低 → 强提示「输入克数校准」或「改用搜索录入」
* **可解释性**：
  * 展示「为何建议校准」（如：无参考物/多食物重叠/置信度低）

## 4.6 MVP 范围（饮食拍照识别）

必须实现：

* **营养价值表 OCR 识别** + 输入克数自动计算热量
* 拍照上传 + 返回候选食物与总热量
* 支持用户确认/修改食物项并保存
* 参考物录入（拳头 + 硬币，仅需一次）
* 快捷模式（pending 记录）+ 后补录转 confirmed

暂不实现（后续迭代）：

* 精细营养素（微量元素/维生素）
* 复杂菜品拆解（火锅/自助多盘极复杂）
* 个性化食物库与自动学习（先用可编辑的结构替代）

## 4.7 权限、隐私与合规（产品侧必须明确）

* **权限**：
  * 相机权限：用于拍照识别
  * 相册权限：用于选择图片识别（可选）
* **用户告知**（首次使用/设置页可查看）：
  * 上传的图片将用于热量估算与饮食记录
  * 用户可删除饮食记录与关联图片（若支持对象存储删除，需要明确删除生效时间）
  * 仅用于产品功能与体验优化（如做训练/评估需单独征得同意）
* **最小化原则**：
  * 允许用户使用「快捷模式」只存档，不强制立刻生成热量
  * 不做公开分享与广场曝光（默认私密）

## 4.8 关键数据指标与埋点（用于评估功能是否值得做）

核心指标：

* 拍照入口点击率（饮食记录入口 → 拍照）
* 识别成功率（返回 items 且可保存）
* 置信度分布（高/中/低）
* 从「快捷模式」转为 confirmed 的转化率（补录率）
* 平均录入耗时（拍照 → 保存）
* 日均饮食记录次数（功能上线前后对比）

建议事件（示例）：

* `diet_photo_open_camera`
* `diet_photo_upload_success` / `diet_photo_upload_fail`
* `diet_photo_analyze_success` / `diet_photo_analyze_fail`
* `diet_record_save_confirmed` / `diet_record_save_pending`
* `diet_record_edit_item_add/remove/update_grams`

## 4.9 验收标准（MVP）

* **流程可走通**：
  * 从记录页拍照 → 出识别结果 → 可编辑 → 可保存为 confirmed
  * 识别失败时 → 可一键保存为 pending（快捷模式）
  * pending 记录可在详情页补录后转为 confirmed
* **数据正确性（结构）**：
  * diet 记录包含：图片 URL、meal_type、mode、status、items、total（confirmed）
  * 修改克数后，单项热量与总热量会更新并保存
* **体验要求**：
  * 首次使用有明确权限请求与用途说明
  * 上传/识别有加载态与失败重试
  * 在弱网/超时场景不丢失照片（至少能进入 pending）

功能：

* 拍照识别
* 热量估算
* 营养分布分析
* AI饮食优化建议

设计重点：

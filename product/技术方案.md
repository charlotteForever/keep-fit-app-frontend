#### 饮食记录（新增：拍照识别热量）

前端需要支持三条链路：

* **营养价值表识别（优先）**：拍照营养成分表 → OCR 识别 → 输入食用克数 → 自动计算 → 保存
* **参考物拍照识别**：拍照（食物+拳头） → 上传 → 返回候选食物/估算热量 → 用户确认/修改 → 保存
* **快捷模式（pending）**：拍照 → 仅存档 → 稍后补录克数/选择食物 → 重算 → 保存确认

关键点：

* **营养价值表优先**：包装食品优先引导用户拍摄营养成分表，数据更准确
* **参考物/对照物**：首次使用时引导录入（拳头 + 硬币，仅需一次），拍照估算时必须同框拍到食物和拳头，用于份量估算稳定性
* **失败兜底**：识别失败或置信度低时，允许一键转快捷模式，避免阻塞用户记录

##### 前端模块拆分（建议）

* `screens/Record/DietModeSelectScreen`：选择识别模式（营养价值表/参考物拍照/快捷模式）
* `screens/Record/DietCameraScreen`：相机拍照/相册选取、权限请求
* `screens/Record/NutritionLabelScreen`：营养价值表 OCR 识别结果、输入克数、计算热量
* `screens/Record/DietAnalyzeScreen`：上传进度、识别请求、loading/重试
* `screens/Record/DietResultScreen`：候选列表、增删食物、编辑克数/份数、保存
* `screens/Record/DietRecordDetailScreen`：pending → confirmed 补录入口
* `screens/Record/ReferenceSetupScreen`：首次参考物录入（拳头 + 硬币）
* `services/diet.ts`：上传/识别/保存相关 API 封装
* `types/diet.ts`：DietRecord/DietItem/Reference/NutritionLabel 等类型定义

##### 状态机（避免边缘状态混乱）

建议把 diet 记录的关键状态明确为：

* `draft_local`：本地草稿（拍完照但还没拿到上传 URL / 网络中断）
* `pending`：已存档（有图片 URL），未确认 items/total
* `confirmed`：已确认（items/total 完整）

本地草稿可用 MMKV 暂存（最多保留 N 条），避免用户在上传/识别过程中切后台导致丢失。

### 2️⃣ AI 对话页

#### 核心架构：Function Calling

AI 对话页的核心是让 LLM 通过 Function Calling 自主决定调用哪些后端工具，而不是硬编码意图分类逻辑。

```
用户消息
    ↓
后端 AI 模块（携带工具定义 + 对话历史 + 用户上下文）
    ↓
LLM 决策：直接回答 or 调用工具？
    ├── 直接回答 → SSE 流式推送给前端
    └── 调用工具 → 执行对应 Service → 结果返回 LLM → 继续流式回答
```

#### 前端模块拆分

* `screens/AI/AIChatScreen`：主对话界面，消息列表 + 输入框
* `screens/AI/ChatMessageItem`：消息气泡（文字 / 操作结果卡片 / 确认卡片）
* `services/ai.ts`：SSE 连接管理、消息发送、流式 token 接收
* `store/chatStore.ts`：对话历史状态管理（Zustand）

#### 流式输出实现（前端）

前端通过 SSE（Server-Sent Events）接收流式响应：

```typescript
// services/ai.ts
const response = await fetch('/ai/chat', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
  body: JSON.stringify({ message, conversationId }),
});

const reader = response.body.getReader();
// 逐 chunk 读取，追加到消息内容
```

SSE 事件类型：
* `token`：LLM 输出的文字 token，追加到当前消息
* `tool_call`：工具调用开始，前端显示"正在查询..."
* `tool_result`：工具执行完成，前端渲染结果卡片
* `done`：本次回复结束

## 4️⃣ AI 对话模块（Function Calling）

### 后端目录结构

```bash
src/modules/ai/
 ├── ai.controller.ts        # POST /ai/chat（SSE 接口）
 ├── ai.service.ts           # 对话主逻辑，驱动 Function Calling 循环
 ├── ai.module.ts
 ├── tools/
 │    ├── tool-registry.ts   # 工具定义列表（传给 LLM 的 tools 参数）
 │    ├── tool-executor.ts   # 根据 LLM 返回的 tool_call 分发执行
 │    └── definitions/
 │         ├── diet.tools.ts
 │         ├── workout.tools.ts
 │         └── trend.tools.ts
 └── dto/
      └── chat.dto.ts
```

### Function Calling 执行流程

```
1. 前端发送消息 → POST /ai/chat
2. 后端组装：system prompt + 对话历史 + 工具定义列表
3. 调用 LLM（stream: true）
4. 解析响应：
   ├── 纯文字 token → SSE event: token → 推给前端
   └── tool_call → SSE event: tool_call → 执行工具
         ↓
      tool-executor 调用对应 Service（DietService / WorkoutService 等）
         ↓
      将工具结果追加到对话历史
         ↓
      再次调用 LLM（携带 tool result）→ 继续流式输出
5. 输出结束 → SSE event: done
```

你的原则：
- 只回答健康相关问题，拒绝无关话题

- 执行写操作（创建记录、修改计划）前，先向用户确认
- 回答简洁，避免冗长说教

用户基本信息：
- 姓名：{nickname}
- 目标：{goal_type}
- 今日热量摄入：{today_calories} kcal / 目标 {calorie_goal} kcal

### SSE 接口设计

```http
POST /ai/chat
Content-Type: application/json
Authorization: Bearer {token}

{
  "message": "帮我看看今天吃了多少热量",
  "conversationId": "uuid"  // 可选，用于多轮对话
}
```

SSE 响应事件流：

```
event: token
data: {"content": "今天"}

event: token
data: {"content": "你一共"}

event: tool_call
data: {"toolName": "get_diet_records", "status": "calling"}

event: tool_result
data: {"toolName": "get_diet_records", "result": {"calories": 1850}}

event: token
data: {"content": "摄入了 1850 kcal"}

event: done
data: {"conversationId": "uuid"}
```

### 饮食拍照识别（新增）

说明：前端是 Expo/RN，建议采用「后端签名直传对象存储」或「后端中转上传」两种模式之一；MVP 可先后端中转，后续切签名直传降低带宽成本。

#### 1) 营养价值表 OCR 识别（优先模式）

```http
POST /diet/nutrition-label/analyze
```

请求：

```json
{
  "image_url": "https://cdn.../xxx.jpg",
  "meal_type": "breakfast"
}
```

返回（示例）：

```json
{
  "status": "ok",
  "confidence": "high",
  "nutrition_per_100g": {
    "calories": 450,
    "protein": 12,
    "fat": 20,
    "carbohydrate": 55,
    "sodium": 800
  },
  "detected_fields": ["calories", "protein", "fat", "carbohydrate"],
  "warnings": []
}
```

用户输入实际食用克数后，前端计算实际摄入：

```javascript
actual_calories = (nutrition_per_100g.calories * grams) / 100
```

#### 2) 获取上传凭证（推荐：直传对象存储）

```http
POST /diet/photo/upload-url
```

返回：

```json
{
  "upload_url": "https://...",
  "file_url": "https://cdn.../xxx.jpg",
  "expires_in": 600
}
```

#### 2) 发起识别（标准模式）

```http
POST /diet/photo/analyze
```

请求：

```json
{
  "image_url": "https://cdn.../xxx.jpg",
  "meal_type": "breakfast",
  "reference": {
    "type": "fist",
    "present_in_image": true
  }
}
```

返回（示例）：

```json
{
  "status": "ok",
  "confidence": "medium",
  "total": { "calories": 520 },
  "items": [
    {
      "name": "炒鸡蛋",
      "candidates": ["炒鸡蛋", "煎蛋"],
      "estimated_grams": 120,
      "calories": 210
    }
  ],
  "warnings": ["no_reference_detected"]
}
```

（约束建议）

* `warnings` 用于前端 UI 提示（例如「未检测到参考物」「菜品重叠」），不影响保存，但应引导用户补录克数校准。
* `confidence` 只做解释与引导，不做强阻断（避免高弃用率）。

#### 4) 创建/更新饮食记录（复用 /record，但建议明确结构）

`/record` 的 `value_json`（diet）建议扩展为：

**营养价值表模式**：

```json
{
  "mode": "nutrition_label",
  "status": "confirmed",
  "meal_type": "breakfast",
  "images": [{ "url": "https://cdn.../nutrition_label.jpg" }],
  "nutrition_per_100g": {
    "calories": 450,
    "protein": 12,
    "fat": 20,
    "carbohydrate": 55
  },
  "consumed_grams": 80,
  "total": {
    "calories": 360,
    "protein": 9.6,
    "fat": 16,
    "carbohydrate": 44
  },
  "source": "nutrition_label_ocr"
}
```

**参考物拍照模式**：

```json
{
  "mode": "reference_photo",
  "status": "confirmed",
  "meal_type": "breakfast",
  "images": [{ "url": "https://cdn.../xxx.jpg" }],
  "confidence": "medium",
  "items": [
    {
      "name": "炒鸡蛋",
      "grams": 120,
      "calories": 210,
      "source": "ai"
    }
  ],
  "total": { "calories": 520 },
  "reference": { "type": "fist", "present_in_image": true }
}
```

**快捷模式（pending）**：

```json
{
  "mode": "quick",
  "status": "pending",
  "meal_type": "breakfast",
  "images": [{ "url": "https://cdn.../xxx.jpg" }],
  "note": ""
}
```

（安全与合规建议）

* 图片 URL 建议为短期可访问（签名 URL）或走 CDN + 鉴权；不要直接暴露永久公网可读原图。
* 记录删除时，若需要删除图片对象，应在后端做异步清理（并对失败重试）。

#### 5) 参考物配置（仅需一次）

```http
PUT /user/profile
```

在用户资料里保存参考物（示例）：

```json
{
  "diet_reference": {
    "fist": {
      "image_url": "https://cdn.../fist.jpg",
      "calibrated": true
    },
    "coin": {
      "image_url": "https://cdn.../coin.jpg",
      "diameter_mm": 25,
      "calibrated": true
    }
  }
}
```

说明：拳头和硬币都需要录入，拳头用于大份量食物对比，硬币用于小份量食物对比。
